Design Patterns Applied — Full Enumeration
1. 🏛️ Facade Pattern
Files: ProductService.java, OrderService.java

Each service class hides the complexity of multiple collaborators (repository + Kafka producer + MapStruct mapper + business rules) behind a single, clean interface. For example, OrderService.createOrder() coordinates UserRepository, ProductRepository, OrderRepository, stock deduction, and Kafka event publishing — the caller sees none of that.

// Caller just calls this one method:
orderService.createOrder(userId, dto);
// Internally: user lookup → product stock check → deductStock → persist → publishKafkaEvent

2. 👁️ Observer / Event-Driven Pattern
Files: DomainEvent.java, OrderEventConsumer.java, UserEventConsumer.java, AuditEventConsumer.java

DomainEvent is the abstract "subject". Kafka topics act as the event bus. OrderEventConsumer, UserEventConsumer, and AuditEventConsumer are the "observers" — they react to events without being coupled to the producer at all.

// Producer publishes and forgets — no knowledge of who listens:
kafkaEventProducer.publish(TOPIC_ORDER_CREATED, order.getOrderNumber(), event);

// Multiple independent consumers each react in their own way:
@KafkaListener(topics = AppConstants.TOPIC_ORDER_CREATED, groupId = GROUP_NOTIFICATION)
public void handleOrderCreated(OrderCreatedEvent event, ...) { ... }

3. 🗃️ Repository Pattern
Files: ProductRepository.java, OrderRepository.java, UserRepository.java, CategoryRepository.java

All data access is abstracted behind JpaRepository interfaces. Business services never write raw SQL — they call expressive methods like findBySku(), searchProducts(), or findByDateRange(). The persistence implementation is fully hidden.

// Expressive domain query — no SQL in business code:
productRepository.searchProducts(categoryId, minPrice, maxPrice, search, pageable);

4. 🗺️ DTO / Mapper Pattern (Data Transfer Object)
Files: ProductMapper.java, OrderMapper.java, UserMapper.java

MapStruct mappers strictly separate the persistence layer (JPA entities with @Version, lazy associations) from the transport layer (DTOs serialised over HTTP). The mapper also supports partial updates via @BeanMapping(nullValuePropertyMappingStrategy = IGNORE).

// Partial update — only non-null DTO fields are applied to the existing entity:
@BeanMapping(nullValuePropertyMappingStrategy = NullValuePropertyMappingStrategy.IGNORE)
void updateEntityFromDto(ProductDto dto, @MappingTarget Product product);

5. 🔌 Adapter Pattern
File: ProductWebServiceImpl.java

ProductWebServiceImpl adapts the Spring Data ProductRepository (a Java interface) to the JAX-WS ProductWebService SOAP contract (XML/WSDL). B2B clients send SOAP XML; internally it delegates to the same JPA repository used by the REST API — zero business logic duplication.

// Adapts JPA result → SOAP-safe XML DTO:
return productRepository.findById(productId)
        .map(this::toSoapResponse)   // ← the adaptation
        .orElseThrow(...);

6. 🎨 Decorator Pattern
File: CorrelationIdFilter.java

Implements both ContainerRequestFilter and ContainerResponseFilter. It transparently wraps every HTTP request/response: injects/generates a X-Correlation-Id header, places it in the SLF4J MDC for log tracing, and echoes it back on the response — all without touching any business code.

// Transparently decorates every request before it reaches a resource:
MDC.put(MDC_KEY, correlationId);
requestContext.setProperty(MDC_KEY, correlationId);

7. ⛓️ Chain of Responsibility Pattern
Files: ResourceServerConfig.java, GlobalExceptionMapper.java

Two manifestations:

Spring Security filter chain — each SecurityFilterChain bean processes a request in order; if a filter passes it, the next one takes over.
GlobalExceptionMapper — exceptions bubble up and are tested in sequence (instanceof ResourceNotFoundException → BusinessValidationException → UnauthorizedException → ConstraintViolationException → fallback), each with its own HTTP status mapping.
// Each exception type is handled independently in sequence:
if (ex instanceof ResourceNotFoundException e)  → 404
if (ex instanceof BusinessValidationException e) → 409
if (ex instanceof UnauthorizedException e)       → 401
if (ex instanceof ConstraintViolationException e) → 400
// fallback                                      → 500

8. 📐 Template Method Pattern
File: ApiResponse.java, PagedResponse.java

Static factory methods define the fixed skeleton of an HTTP response (success flag, timestamp, data/error payload) while leaving the variable parts (the actual data T, the message) to the caller. Every resource uses the same envelope consistently.

// Fixed structure — caller only fills the variable part (the data):
ApiResponse.success(productDto)
ApiResponse.success(productDto, "Product created successfully")
ApiResponse.error("Validation failed", violations)

// PagedResponse auto-computes totalPages and last flag:
PagedResponse.of(content, page, size, totalElements);

9. 🎯 Strategy Pattern
File: KafkaEventProducer.java

The producer class is topic-agnostic. The calling service supplies the topic (the "strategy") at runtime. This means the same publish() method handles order.created, product.updated, user.registered, etc., without any conditional branching inside the producer itself.

// The caller picks the strategy (topic); the producer handles transport uniformly:
kafkaEventProducer.publish(TOPIC_ORDER_CREATED,   order.getOrderNumber(), event);
kafkaEventProducer.publish(TOPIC_PRODUCT_UPDATED, productId,              event);
kafkaEventProducer.publish(TOPIC_USER_REGISTERED, username,               event);

10. 🏗️ Builder Pattern
Files: Order.java, Product.java, all event classes, ApiResponse, PagedResponse

Lombok's @Builder is applied to all entities, DTOs, and event objects. This gives readable, type-safe, step-by-step object construction without telescoping constructors.

// Readable, safe, no constructor overloading:
Order order = Order.builder()
        .orderNumber(generateOrderNumber())
        .user(user)
        .shippingAddress(dto.getShippingAddress())
        .build();

11. 🧩 Aggregate Root Pattern (DDD)
File: Order.java

Order is the aggregate root — it owns its OrderItem collection and enforces all invariants internally. Business logic like addItem() (which recalculates the total), recalculateTotal(), and isCancellable() lives inside the entity, not scattered in service code.

// Business logic inside the aggregate root — not in the service:
public void addItem(OrderItem item) {
    items.add(item);
    item.setOrder(this);
    recalculateTotal();   // ← invariant enforced automatically
}

public boolean isCancellable() {
    return status == OrderStatus.PENDING || status == OrderStatus.CONFIRMED;
}

How Profiles Work in This Project
Spring Boot's profile system means only one set of files changes between environments — the application.yml base always loads first, then the active profile file overrides only what differs.

application.yml           ← shared base (port, JPA batch settings, Flyway locations)
    ↑ merged with
application-local.yml     ← localhost URLs, show-sql=true, DEBUG logging
application-docker.yml    ← container hostnames, INFO logging, pool=20
application-h2.yml        ← H2 in-memory, Kafka disabled, no Gateway needed

The key guard — KafkaConfig.java
@ConditionalOnProperty(name = "spring.kafka.enabled", havingValue = "true", matchIfMissing = true)
